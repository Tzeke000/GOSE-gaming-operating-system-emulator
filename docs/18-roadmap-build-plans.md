## Desktop / windowed multitasking for the GOSE web shell (window manager + memory-aware widget↔window continuum)  
**Effort:** large

**Summary:** GOSE's shell is a single GTK window (kiosk.py) hosting ONE WebKit2GTK WebView that loads static HTML pages from /userdata/gose-ui, navigated by full-page location.href swaps; widgets are draggable divs in gose-home.html persisted to localStorage; native apps/games launch as separate X windows ON TOP of the kiosk via _spawn(). Build a window manager as a new desktop shell page (gose-desktop.html) that hosts in-shell "apps" (the existing HTML pages, unchanged, loaded into iframes) as movable/resizable/maximizable windows using WinBox.js (Apache-2.0, ~6KB, zero-dep, native iframe/minimize/maximize/focus/z-index + full event API + mount/unmount). Widgets and app-windows become ONE continuum: a widget is a small "docked" window; MAXIMIZE promotes it to a full window (same WinBox instance, just maximize()); and "acting out" of it SUSPENDS it — the iframe is torn down (src→about:blank, element removed) so its DOM/JS heap is GC'd, and a compact state token (route + scroll + form state) is persisted so resume re-creates the iframe restored. A new server-side window registry (/wm/*) tracks open windows, their memory state (live/suspended), and an LRU/RAM-budget policy that auto-suspends the least-recently-focused windows when free RAM drops. Native apps/games are surfaced as window tiles in the SAME switcher by enumerating X windows (wmctrl, already an xdotool-class dep) and raising/closing them through the existing /game/exit + new /wm/native endpoints — so Alt-Tab walks both web windows and native windows. Phase 2 upgrades the memory-freeing from "iframe teardown" (frees JS/DOM heap only, single web process) to TRUE per-window OS RAM release by refactoring kiosk.py into a GtkStack of N WebKitWebViews under PROCESS_MODEL_MULTIPLE_SECONDARY_PROCESSES, where suspend calls webkit_web_view_terminate_web_process() to kill that window's web process outright. Phase 1 (WinBox + iframe teardown) is shippable on the current kiosk with no GTK changes; phase 2 is the optional compositor that makes "frees memory like closing an app" literally true at the OS level.

**Reuse (don't reinvent):**
- WinBox.js (https://github.com/nextapps-de/winbox , Apache-2.0, ~6KB zero-dep) — the window-manager core: draggable/resizable/maximize/minimize/fullscreen, iframe content via url/setUrl(), full event API (onfocus/onblur/onminimize/onmaximize/onclose/onresize), z-index/focus via WinBox.stack()/focus(), and mount()/unmount() to detach body content. Maps 1:1 onto the widget↔window continuum and the suspend/resume hooks. Reuse instead of hand-rolling drag/resize.
- WebKitGTK terminate_web_process() (https://webkitgtk.org/reference/webkit2gtk/2.40.4/method.WebView.terminate_web_process.html) — the exact phase-2 primitive: kill ONE window's web process to free its RAM at the OS level; next load spawns a fresh process. This is what makes 'suspends and frees memory like closing an app' literally true.
- WebKit ProcessModel.MULTIPLE_SECONDARY_PROCESSES + WEBKIT_CACHE_MODEL_DOCUMENT_VIEWER (https://valadoc.org/webkit2gtk-4.0/WebKit.ProcessModel.MULTIPLE_SECONDARY_PROCESSES.html , https://webkitgtk.org/reference/webkit2gtk/2.5.1/WebKitWebView.html) — one web process per window WebView + the low-memory cache model for an app shell; the compositor foundation for true per-window RAM accounting.

**Risks:**
- Iframe teardown frees JS/DOM heap but NOT the single shared web process's full footprint — phase-1 'free memory' is real but partial. Only phase-2 (terminate_web_process per-view) gives OS-level reclamation. Be honest in docs/Task-Manager about which tier is active; don't claim full RAM-free until phase 2 ships (verify-before-asserting).
- Same-origin iframes: all pages are served from one origin (localhost:8780), so postMessage state capture and DOM access work — but if any app loads a cross-origin URL (Store→Chrome is a NATIVE app, fine; but a future in-frame web view) the state bridge breaks; keep cross-origin content as native windows, not iframes.
- Native window control via wmctrl requires wmctrl present in the Batocera image; if absent, fall back to xdotool (already used) for search/raise and pkill/_GAME_PATS for close. Verify wmctrl availability in the VM before depending on it.
- Phase-2 compositor is a real architectural lift: WinBox chrome draws in DOM but per-window WebViews are GTK widgets — reconciling WinBox's HTML titlebars with GTK-positioned WebViews means either GTK-native chrome or a transparent-overlay trick. Risk of input/z-order glitches; this is why phase 2 is gated and phase 1 (pure web, single WebView) ships first.
- Resume-restore fidelity: some app pages have in-flight async state (terminal scrollback, a running splice job) that a route+scroll token won't fully restore. Mark such apps suspendable:false in the manifest (keep them live) or persist their state server-side; don't silently lose work.

- Performance: many live WinBox iframes on the VM's GPU could stutter; the LRU/RAM policy must be conservative (small live-window cap, e.g. 3-4) so the device stays smooth — tune against /status.json on the actual VM.

## Multiplayer (human + AI in the same game; AI-vs-AI; co-op) over LAN/Bluetooth — join/host flow, control arbitration, AI input delivery to the emulator  
**Effort:** medium

**Summary:** GOSE already has the load-bearing pieces: the device agent (port 8731) injects gamepad input by creating ONE uinput device ("GOSE Virtual Gamepad", agent/gose_agent/capabilities/input.py:82-157), the AI-hub presence registry + grant/tier store already exist in gose_vm_server.py (ai_join/heartbeat/leave + ai_grant/revoke, lines 1051-1145) and are surfaced by the working AI Players UI (gui/mockup/gose-ai.html) and per-op tiering in the agent server (server.py:27-38). What multiplayer needs is NOT new input plumbing — it's (1) a way to make the AI's virtual pad land on a DETERMINISTIC, separate player slot from the human's real controller so both occupy the same running emulator, and (2) a session/arbitration layer that tracks who owns which player port and gates the agent's input.* ops by that ownership. The cleanest, reuse-first design: the AI's uinput pad is named/identified so RetroArch's existing device-reservation feature pins it to a fixed port (e.g. P2), the human's physical pad stays P1; a small "match session" object in the VM server records seat assignments (P1=human, P2=AI) and the agent enforces a per-connection player_slot so an AI can only drive its own seat. AI-vs-AI = two virtual pads (P1+P2) both agent-driven; co-op (both on same side) = same machinery with both seats reachable. For two-machine / remote AI-vs-AI we reuse RetroArch's own netplay (netplay_request_device_pN) rather than inventing sync. LAN is the agent's existing TCP transport; "Bluetooth" reuses the existing BT radio plumbing (RFCOMM/PAN) as just another transport to the same agent port — no protocol change. This is a medium build: most code is a thin SeatManager + reservation config wiring + UI; the risky part is verifying multi-pad slotting on the real Batocera guest.

## OS fundamentals — Accessibility (text scale, high-contrast, colorblind palettes/correction) for the GOSE shell  
**Effort:** medium

**Summary:** Among the six gaps, Accessibility is the highest-value pick for GOSE's stated goal of becoming a downloadable/Steam-distributable app: it's a distribution prerequisite (docs/17 §E Steam path + §I "safe out-of-box defaults audit"), it benefits every user on a small handheld screen, and — critically — it is almost pure reuse of the theming pattern GOSE already ships (themes.css + `<html data-theme>` + localStorage), so it needs essentially zero new backend surface and is low-risk versus VPN/hotspot (heavy/fragile Windows+Batocera plumbing) or session-switching (blocked on a real password/PIN backend that's still a localStorage stub). The design adds three orthogonal axes — Text scale (root font-size in rem via a `--ui-scale` CSS variable), High contrast (a `data-contrast="high"` attribute that overrides theme tokens to WCAG-AAA-contrast values + thickens focus rings, mirroring the existing data-theme mechanism), and Colorblind support (two sub-modes: a "friendly palette" that swaps accent/status colors to a colorblind-safe set, and an optional full-screen daltonization *correction* filter via an inline SVG feColorMatrix referenced by `filter:url(#cb-deut)` on the root) — plus an OS-wide "Reduce motion" toggle that gates the existing animated wallpapers/nebula canvases. All settings persist in localStorage (matching gose-theme/gose-wp) and are applied by a single new shared script `assets/a11y.js` that every page already-loads-alongside platform.js/cursor.js includes, so one include lights up accessibility across the entire shell. A tiny optional backend endpoint persists the prefs server-side so they survive a localStorage clear and are readable per-account later, but the feature works fully without it.

**Reuse (don't reinvent):**
- GOSE's own themes.css + `<html data-theme>` + localStorage pattern (gui/mockup/assets/themes.css, gose-settings.html applyRow/initPersist) — the high-contrast and colorblind-palette modes are just additional token sets on the SAME mechanism; reuse it instead of inventing a settings/persist layer.
- libDaltonLens SVG CVD filters — https://github.com/DaltonLens/libDaltonLens/blob/master/svg/cvd_svg_filters.html (public domain) — accurate ready-made feColorMatrix filters (linearRGB) for protan/deutan/tritan; copy the matrices for the correction/simulation filter block in a11y.js.
- hail2u/color-blindness-emulation — https://github.com/hail2u/color-blindness-emulation (CC0) — drop-in filters.svg covering 8 CVD types; alternative/source for the feColorMatrix values, license-clean for redistribution in a downloadable OS.
- Okabe-Ito colorblind-safe qualitative palette — https://jfly.uni-koeln.de/color/ (public reference) — the 8-color set to remap GOSE's --accent + status (#3fe0a0 success / #ff5a6e error) for the 'Friendly palette' mode.

## Security for shipping: sandbox untrusted/3rd-party AIs, per-user data isolation, safe-out-of-box defaults audit (GOSE distribution hardening)  
**Effort:** large

**Summary:** GOSE today enforces AI permission *tiers* (observe/play/admin) at the protocol boundary in agent/gose_agent/server.py + the VM server's ai_grants/ai_tokens stores, but there is NO actual sandbox: an `admin`-tier (or any code path that reaches `system.run`/`term_exec`/`run_file`) executes arbitrary shell as root on a Batocera/ROCKNIX box where the root FS is a read-only squashfs and everything runs as a single root uid with a shared /userdata. So the tier check is a policy gate with no enforcement floor, there is no per-user OS isolation (one /userdata, one uid, one /userdata/home shared by all human+AI accounts), and the grant/token stores are plaintext JSON. The plan adds three independent-but-layered hardening tracks that REUSE existing tools rather than reinvent: (1) Sandboxing — wrap every shell/exec capability (agent system.run/service, VM term_exec/run_file/launch_app) in a Landlock-based confiner (landrun CLI, with bubblewrap as the namespace fallback) so even an `admin` AI is confined to an allow-listed FS view + no-network-by-default + a command allow/deny policy keyed to its tier and identity; downgrade the agent's deny-by-default from "unmapped op needs admin" to "unmapped op is REFUSED unless explicitly listed," and split today's monolithic `admin` into `admin` (settings/power/store) vs a separate, rarely-granted `shell` scope. (2) Per-user isolation — give every GOSE account (human or AI) its own uid + chrooted/bind-mounted home root under /userdata/users/<uid>, run that account's sandboxed processes under that uid, and confine the Files app + term to that root via the existing _safe() pattern extended per-session. (3) Safe-defaults audit — a scripted, repeatable boot-time + CI audit (OWASP Secure-by-Default / IoT least-privilege checklist) that asserts: agent binds loopback-only unless a token is set + remote explicitly enabled, no AI defaults above observe, shell scope is off by default, stores are 0600/encrypted, no demo/back-compat admin tokens ship, and the VM server stays loopback-bound. NOTE/CORRECTION for the owner: docs/16 cites "Capframe" as the macaroon library to adopt — I could not verify that project exists; the real, verifiable equivalent is pymacaroons (+ ed25519 via PyNaCl) for the attenuation-only / holder-of-key tokens, so the plan substitutes those. Research/design only; no files edited.

**Reuse (don't reinvent):**
- landrun — https://github.com/Zouuup/landrun (Apache-2.0): wrap each shell/exec call (system.run, term_exec, run_file, emulator launch) in an unprivileged Landlock sandbox — no root, no containers, single static Go binary; --ro/--rw/--rox/--bind flags give per-tool FS + network confinement (network rules need kernel 6.7+).

- Landlock kernel LSM / py-landlock — https://github.com/theori-io/py-landlock (and https://landlock.io): if we want the confinement in-process (Python self-restriction) instead of an external binary, this gives a self-sandboxing fallback for the VM server.
