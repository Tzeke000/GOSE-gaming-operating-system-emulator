# system-splash — Batocera early-boot splash replacement (rootfs)

`boot-logo.png` (16:9) and `boot-logo-4x3.png` (4:3) replace Batocera's stock
logo at `/usr/share/batocera/splash/` so the FIRST thing the framebuffer shows
(init script `S03system-splash`, fbv, ~6s into boot — right after virtio-gpu
registers fb0) is the GOSE crystal, not the Batocera brand.

Generated by `../_make_splash.py` from `gui/mockup/assets/brand/gose-crystal.png`
— same render as `/userdata/splash/gose-splash.png` and the kiosk's
`gose-boot.html`, so the boot is brand-consistent end-to-end.

## Why these are NOT in `../splash/`

`build-gose-pc.sh` rsyncs `../splash/` wholesale into `/userdata/splash/`, and
Batocera's S28 splash service picks a **random** image from that directory —
extra files there cause random off-brand splashes. These two files target the
**rootfs** instead, which the userdata-only build step cannot reach.

## How to apply

* **Live VM (no rebuild):** copy both files over
  `/usr/share/batocera/splash/boot-logo*.png` (the running rootfs overlay is
  writable), then run `batocera-save-overlay` so the change survives reboot
  (persists into `/boot/boot/overlay`). Keep `.bak` copies beside the originals.
  Done on the dev VM 2026-06-07.
* **Image bake (TODO for the build):** `build-gose-pc.sh` only mounts the
  userdata partition; shipping this in a fresh image needs either (a) a
  pre-populated `/boot/boot/overlay` ext4 file added to the boot partition, or
  (b) repacking the rootfs squashfs (`/boot/boot/batocera`). Until one of those
  is wired into the build, a fresh image shows the stock Batocera logo for the
  ~1s of S03 and goes crystal from S28 onward.
